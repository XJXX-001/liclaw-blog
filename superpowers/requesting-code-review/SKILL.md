# Requesting Code Review

## 触发条件

完成实现，需要其他代理或人类审查时使用。

## 准备审查材料

### 1. 变更摘要

```markdown
## [功能名称] 代码审查请求

### 变更类型
- [ ] 新功能
- [ ] 修改
- [ ] 修复
- [ ] 重构

### 简要描述
一句话描述变更

### 动机
为什么要做这个变更？
```

### 2. 变更详情

```markdown
### 新增文件
- `src/new-file.py`

### 修改文件
- `src/existing-file.py:12-45`
- `tests/test-file.py:1-30`

### 删除文件
- （如有）
```

### 3. 测试结果

```markdown
### 测试
- 单元测试：X 个通过
- 集成测试：（如适用）
- 手动测试：（如适用）

### 覆盖
- 覆盖率：X%
- 边界条件测试：（是否覆盖）
```

### 4. 自审问题

```markdown
### 自审发现的问题
- 问题 1（已修复）
- 问题 2（已修复）

### 需要审查者注意的点
- 关注点 1
- 关注点 2
```

## 派发审查

### 派发给子代理

使用 `sessions_spawn` 派发代码审查代理：

**审查代理提示词应包含：**
1. 变更摘要
2. 完整 diff 或变更文件列表
3. 审查标准
4. 需要检查的要点

### 审查标准

```
1. 正确性
   - 代码逻辑正确？
   - 测试覆盖够？

2. 安全性
   - 有安全漏洞？
   - 输入验证？

3. 可维护性
   - 代码清晰？
   - 命名合理？

4. 性能
   - 有性能问题？
   - 资源使用合理？

5. 一致性
   - 遵循项目规范？
   - 与既有代码风格一致？
```

## 审查结果处理

| 结果 | 处理 |
|------|------|
| ✅ 通过 | 可以合并 |
| ⚠️ 小问题 | 修复后再次审查 |
| ❌ 大问题 | 需要大改，重新实现 |

## 输出格式

```markdown
## 代码审查报告

### 审查者
[代理名/用户名]

### 总体评价
[通过/有问题/需要大改]

### 优点
- 优点 1
- 优点 2

### 问题

#### 问题 1（[严重/中等/轻微]）
描述：...
建议：...

#### 问题 2（[严重/中等/轻微]）
描述：...
建议：...

### 建议
[其他改进建议]

### 结论
✅ 可以合并 / ⚠️ 修复后合并 / ❌ 需要重新实现
```
