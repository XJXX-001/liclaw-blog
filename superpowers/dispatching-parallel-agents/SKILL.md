# Dispatching Parallel Agents

## 适用条件

**任务必须满足：**
1. 独立 — 任务之间没有依赖
2. 可并行 — 同时执行不会冲突
3. 隔离上下文 — 子代理不需继承主会话上下文

**典型场景：**
- 多个独立功能的实现
- 多个文件的独立修改
- 并行测试多个模块
- 同时调研多个方案

## 何时不用

- 任务相互依赖 → 顺序执行
- 需要共享状态 → 共享有问题
- 人类需要交互 → 顺序更安全

## 工作流程

```
识别独立任务
    ↓
评估并行可行性
    ↓
规划派发策略
    ↓
同时派发所有子代理
    ↓
收集结果
    ↓
合并整合
```

## 派发策略

### 策略 1：全部并行

适用于：任务数量少（≤5），每个任务大

```
派发 A → │
派发 B → │ 同时执行
派发 C → │
```

### 策略 2：分批并行

适用于：任务数量多，需要控制并发

```
批次 1：
派发 A → │
派发 B → │ 同时执行
派发 C → │

批次 1 完成

批次 2：
派发 D → │
派发 E → │ 同时执行
派发 F → │
```

### 策略 3：主控 + 工作

适用于：需要协调结果

```
主控代理（你）
    ↓ 派发任务
子代理 A ←─┐
子代理 B ←─┼─ 同时执行
子代理 C ←─┘
    ↓ 收集结果
主控整合
```

## 子代理指令模板

每个派发的子代理需要清晰的指令：

```markdown
## 任务：[任务名称]

### 上下文
[项目背景、技术栈、约束]

### 任务描述
[具体要做什么]

### 交付要求
- [ ] 交付物 1
- [ ] 交付物 2

### 约束
- 使用 [技术栈]
- 遵循 [编码规范]
- 不超过 [时间/资源限制]

### 完成后
汇报：
1. 完成状态
2. 变更摘要
3. 任何问题
```

## 模型选择

| 任务类型 | 模型 | 理由 |
|----------|------|------|
| 独立实现 | 快速便宜模型 | 任务明确，无需强推理 |
| 需要设计 | 标准模型 | 需要判断 |
| 复杂整合 | 最强模型 | 整合需要深度思考 |

## 状态收集

派发后，等待所有子代理完成：

```markdown
派发状态：
- [ ] 任务 A — 派发中...
- [ ] 任务 B — 派发中...
- [ ] 任务 C — 派发中...

等待所有子代理完成...
```

## 结果整合

收集完成后，整合所有结果：

```markdown
## 并行执行完成

### 任务 A — ✅ 完成
变更：`src/a.py`
测试：5 个通过

### 任务 B — ✅ 完成
变更：`src/b.py`
测试：3 个通过

### 任务 C — ⚠️ 有问题
问题：边界条件未覆盖
处理：已修复

### 整合结果
- 所有任务完成
- 需要在主会话整合测试
```

## 常见问题

| 问题 | 处理 |
|------|------|
| 一个子代理失败 | 评估：重派 or 手动 or 跳过 |
| 结果冲突 | 在主会话仲裁 |
| 部分超时 | 检查已完成结果，评估未完成 |
| 依赖共享资源 | 改为顺序执行 |

## 限制

- 并行数量受限于模型配额
- 建议一次最多 5 个并行
- 复杂协调任务不适合并行

## 与 subagent-driven-development 的区别

| 方面 | 并行派发 | Subagent-Driven |
|------|----------|-----------------|
| 任务关系 | 独立 | 可独立可依赖 |
| 执行方式 | 同时 | 顺序 |
| 协调 | 只需收集 | 每步审查 |
| 速度 | 最快 | 中等 |
| 风险 | 结果整合复杂 | 顺序可控 |

## Red Flags

**不要并行派发：**
- ❌ 相互依赖的任务
- ❌ 需要共享状态
- ❌ 人类需要逐步确认
- ❌ 超过 5 个任务（控制复杂度）
