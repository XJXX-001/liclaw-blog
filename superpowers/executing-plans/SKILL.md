# Executing Plans

## 适用场景

与 `subagent-driven-development` 的对比：

| 场景 | 推荐 |
|------|------|
| 在当前会话执行，任务独立 | `executing-plans` |
| 并行派发子代理到其他会话 | `subagent-driven-development` |
| 任务相互依赖 | `executing-plans`（顺序安全） |
| 需要频繁人类交互 | `executing-plans` |

## 核心原则

**顺序执行 + 检查点 = 稳定可控**

每个任务完成后停下来，让人类审查，再继续。

## 工作流程

```
读取计划 → 提取任务 → 为每个任务创建 TodoWrite

Per Task：
    ↓
执行 Task 所有 Steps
    ↓
完成前自审
    ↓
检查点暂停 → 人类审查
    ↓
通过 → 下一个任务
    ↓
不通过 → 修复 → 重新审查
```

## 执行步骤

### Step 1: 读计划

读取完整计划文件一次，提取所有任务。

### Step 2: 创建 TodoWrite

```markdown
## 实施计划：[功能名]

- [ ] Task 1: [名称]
- [ ] Task 2: [名称]
- [ ] Task 3: [名称]
```

### Step 3: 执行每个 Task

对于每个 Task：

```
执行所有 Steps
    ↓
完成后自审：
    - 测试通过？
    - 代码质量 OK？
    - 无调试代码残留？
    ↓
检查点暂停，等人类审查
```

### Step 4: 人类审查

向人类展示：

```markdown
## Task N 完成

**做了什么：**
- 简要描述

**变更文件：**
- `file1.py`
- `file2.py`

**测试结果：**
- X 个测试通过

请审查后说"继续"或反馈问题。
```

### Step 5: 继续或修复

- "继续" → 下一个任务
- 反馈问题 → 修复 → 重新提交审查

## Task 内执行顺序

每个 Task 内按顺序执行 Steps：

```
红：写失败测试
    ↓
绿：写最小实现
    ↓
重构：改善（如需要）
    ↓
自审
    ↓
提交
    ↓
检查点
```

## 自审清单

完成 Task 后自审：

```
[ ] 测试失败（红）？
[ ] 测试通过（绿）？
[ ] 代码质量 OK？
[ ] 提交信息清晰？
[ ] 无调试代码残留？
[ ] 遵循项目规范？
```

发现问题 → 修复 → 再提交

## 与 Subagent-Driven 的区别

| 方面 | Executing Plans | Subagent-Driven |
|------|-----------------|-----------------|
| 执行位置 | 当前会话 | 子代理会话 |
| 并行 | 否 | 是 |
| 上下文污染风险 | 低 | 极低 |
| 人类交互 | 每个 Task 后 | 可配置 |
| 速度 | 较慢 | 较快 |

## 何时选择 Executing Plans

✅ 任务相互依赖，必须顺序执行
✅ 需要频繁人类确认
✅ 任务简单，不需要子代理开销
✅ 调试阶段，每步都需要人类指导

❌ 任务独立，想并行加速 → 用 subagent-driven
❌ 想最小化人类交互 → 用 subagent-driven
