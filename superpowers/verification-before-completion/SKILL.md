# Verification Before Completion

## 触发条件

**任何任务完成前，必须执行此检查。**

## 检查清单

### 1. Spec 覆盖检查

逐条过 spec，确保每条都有对应实现：

```
[ ] Spec 第 1 条 — 对应实现：XXX
[ ] Spec 第 2 条 — 对应实现：XXX
[ ] Spec 第 3 条 — 对应实现：XXX
[ ] 有遗漏吗？列出
```

### 2. 测试覆盖检查

```
[ ] 核心功能有测试？
[ ] 边界条件有测试？
[ ] 错误处理有测试？
[ ] 测试都通过？
```

### 3. 代码质量检查

```
[ ] 代码遵循项目规范？
[ ] 命名清晰？
[ ] 没有明显 bug？
[ ] 没有调试代码残留？
```

### 4. 文档检查

```
[ ] 需要更新的文档已更新？
[ ] 复杂逻辑有注释？
[ ] API 变更有说明？
```

### 5. 提交检查

```
[ ] 提交信息清晰？
[ ] 提交粒度合理？（不大不小）
[ ] 相关文件都提交了？
```

## 执行顺序

**按顺序执行，发现问题就停：**

```
Spec 覆盖
    ↓ 发现问题
修复 → 重新验证 Spec

Spec OK
    ↓
测试覆盖
    ↓ 发现问题
修复 → 重新验证测试

测试 OK
    ↓
代码质量
    ↓ 发现问题
修复 → 重新验证质量

质量 OK
    ↓
文档
    ↓ 发现问题
更新文档

文档 OK
    ↓
提交
```

## 常见问题

| 问题 | 处理 |
|------|------|
| Spec 覆盖不完整 | 回到 writing-plans，补上遗漏任务 |
| 测试不足 | 用 TDD 补写测试 |
| 代码质量差 | 重构，或派发子代理修复 |
| 文档过时 | 更新文档，或标记 TODO |

## 输出格式

```markdown
## 验证报告

### Spec 覆盖
- [x] Spec 1.1 — 已实现
- [x] Spec 1.2 — 已实现
- [ ] Spec 2.1 — **遗漏**，需要补

### 测试
- [x] 核心功能测试 5/5 通过
- [ ] 边界条件测试 — **不足**，需要补

### 代码质量
- [x] 通过代码审查

### 文档
- [x] README 已更新

### 下一步
需要补完 Spec 2.1 的测试，然后重新验证
```

## 完成标志

所有检查项 ✅ 后，任务才算完成。

---

**简单任务**：检查清单可以简短，但必须有。

**复杂任务**：详细记录每项检查结果。
