# Brainstorming

## 触发条件

用户提出任何"要做XX"、"怎么做XX"、"帮我实现XX"时，先激活此 skill。

**不要急着给答案。** 先帮助用户明确真正的问题。

## 工作流程

### 第一步：理解真正的问题

用这些问题引导：

```
"你真正想要解决的是什么？"
"这个问题的本质是什么？"
"最终效果是什么样的？"
"怎么才算'做好了'？"
```

### 第二步：约束条件

```
"有什么限制？"
"技术栈限制？"
"时间限制？"
"资源限制？"
```

### 第三步：探索方案

```
"你考虑过哪些方案？"
"为什么排除了一些方案？"
"有没有参考案例？"
```

## 输出：Spec Document

brainstorming 的输出是一份 **Spec Document**，格式如下：

```markdown
# [功能名称] 规格说明

## 目标
一句话描述真正要解决的问题

## 约束条件
- 技术限制
- 时间限制
- 其他约束

## 方案设计
[已确定的方案，1-3 个选项]

## 验收标准
- 标准 1
- 标准 2
- 标准 3

## 下一步
- [ ] 进入 writing-plans 制定实施计划
- [ ] 或直接执行（简单任务）
```

## Spec 粒度

**独立子系统 → 独立 spec**
- 如果需求涉及多个独立子系统，每个子系统单独出 spec
- 这样可以并行工作

**复杂系统 → 先拆解**
- 不要试图一个 spec 覆盖所有
- 先确定整体架构，再逐个子系统 spec

## 常见陷阱

**不要做的事：**
- ❌ 直接开始写代码
- ❌ 假设自己理解了需求
- ❌ 给用户你认为他们需要的东西
- ❌ 跳过 spec，直接做简单计划

**要做的事：**
- ✅ 问清楚真正的问题
- ✅ 确认验收标准
- ✅ 输出清晰的 spec
- ✅ 让用户确认后再继续

## 决策树

```
需求涉及多个独立子系统？
    ↓ 是
拆成独立 spec，分别或并行处理

需求复杂，需要架构设计？
    ↓ 是
先架构讨论，再详细 spec

需求简单明确？
    ↓ 是
快速 spec + 直接执行 或 进入 writing-plans
```

## 完成标志

当你输出完整的 Spec Document 并获得用户确认后，此 skill 完成。

---

**使用示例：**

```
用户：帮我建一个博客

你：正在使用 brainstorming skill...

[分析需求：什么类型的博客？目标用户？功能范围？]

Spec 输出：
# 个人博客系统 规格说明

## 目标
为个人作者提供轻量级博客系统

## 约束条件
- 静态生成（可选 SSR）
- 简单部署
- 基础评论功能

## 验收标准
- 可以发布文章
- 可以分类标签
- 可以被搜索引擎收录

[用户确认后]
→ 进入 writing-plans 制定实施计划
```
