# Systematic Debugging

## 核心原则

**不要盲目试错。系统化定位问题。**

## 调试循环

```
复现 → 定位 → 修复 → 验证

复现：稳定复现问题
定位：找到根本原因
修复：最小改动修复
验证：确认修复有效
```

## Step 1: 复现问题

**首要任务：稳定复现 bug。**

```
能稳定复现吗？
    ↓ 否
检查：环境差异？数据差异？时序问题？
    ↓
找到复现条件
```

**复现信息要包含：**
- 完整错误信息
- 复现步骤
- 环境/上下文
- 期望行为 vs 实际行为

## Step 2: 定位问题

### 二分查找法

```
日志/输出在 A 点正常，B 点异常
    ↓
A 和 B 中间加检查点
    ↓
确认问题在前半段还是后半段
    ↓
递归，直到精确到问题代码
```

### 常见问题模式

| 症状 | 可能原因 | 检查点 |
|------|----------|--------|
| 空指针 | 未初始化、API 返回 null | 初始化、API 响应 |
| 超时 | 死循环、网络慢、查询慢 | 循环条件、API 延迟 |
| 数据错 | 类型转换、赋值错误 | 数据转换逻辑 |
| 内存涨 | 内存泄漏、缓存无限增长 | 缓存大小、对象引用 |

### 防御深度

**多层检查，每层捕获不同问题：**

```
用户输入 → 层1验证 → 业务逻辑 → 层2验证 → 数据层 → 层3验证
```

## Step 3: 修复问题

**最小改动原则：**
- 改最少的代码
- 不引入新问题
- 不做大重构

**修复后要自问：**
- 这个修复会引入新 bug 吗？
- 这个修复会影响其他功能吗？
- 测试覆盖够吗？

## Step 4: 验证

**验证修复有效：**
1. 重新运行导致 bug 的场景
2. 确认不再出现
3. 运行相关测试
4. 边界条件也测试

**回归测试：**
- 运行所有测试
- 确认没有破坏其他功能

## 调试技术

### 添加日志

```python
# 在可疑点添加日志
logger.debug(f"Step A: value={value}")

# 追踪执行路径
def function():
    logger.debug(f"Enter function, args={args}")
    result = do_something()
    logger.debug(f"Result: {result}")
    return result
```

### 条件等待（异步问题）

```python
# 不好：固定等待
time.sleep(5)  # 有时不够，有时浪费

# 好：条件等待
start = time.time()
while not condition():
    if time.time() - start > TIMEOUT:
        raise TimeoutError()
    time.sleep(0.1)
```

### 隔离问题

```
问题在哪个层？
    ↓
隔离该层测试
    ↓
确认问题在该层
    ↓
缩小范围到具体代码
```

## 常见错误

| 错误 | 问题 |
|------|------|
| 修 bug 没测试 | 可能引入新 bug |
| 修表面不修根因 | 同 bug 换个方式出现 |
| 过度工程 | 小 bug 大改动 |
| 不记录复现条件 | 下次还会在同样地方挂 |

## 自检清单

修复完成后：

- [ ] 问题在相同条件下稳定复现？
- [ ] 找到了根本原因？
- [ ] 修复是最小改动？
- [ ] 相关测试都通过？
- [ ] 没有回归？

## 升级条件

这些问题需要升级给人类：

- 需要修改架构
- 需要改第三方依赖
- 涉及安全/隐私问题
- 调试时间超过预期
