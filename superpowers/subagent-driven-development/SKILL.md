# Subagent-Driven Development

## 核心原则

**每个任务派发一个全新子代理 + 两阶段 review = 高质量、快迭代**

为什么用子代理：
- 隔离上下文，不污染
- 精确构建指令，子代理专注任务
- 主会话保持清晰，专注协调

## 工作流程

```
读取计划 → 提取所有任务 → 为每个任务创建 TodoWrite

Per Task：
    ↓
派发实现子代理（提供完整任务上下文）
    ↓
子代理问问题？ → 回答问题，提供上下文
    ↓
子代理实现、测试、提交、自审
    ↓
派发 Spec 审查子代理 → 合规？
    ↓ 否 → 修复 → 重新审查
    ↓ 是
派发代码质量审查子代理 → 通过？
    ↓ 否 → 修复 → 重新审查
    ↓ 是
标记任务完成

所有任务完成后 → 最终代码审查 → 分支收尾
```

## 模型选择

**使用能完成任务的最弱模型，节省成本：**

| 任务类型 | 模型 | 示例 |
|----------|------|------|
| 机械实现 | 快速便宜模型 | 单函数、清晰 spec、1-2 文件 |
| 整合判断 | 标准模型 | 多文件协调、模式匹配、调试 |
| 架构设计 | 最强模型 | 需要判断力、广泛理解 |

**信号：**
- 1-2 文件 + 完整 spec → 便宜模型
- 多文件 + 整合问题 → 标准模型
- 需要设计判断 → 最强模型

## 子代理状态处理

子代理报告四种状态：

| 状态 | 含义 | 处理 |
|------|------|------|
| DONE | 完成 | 进入 spec 审查 |
| DONE_WITH_CONCERNS | 完成但有疑虑 | 先看疑虑，再决定 |
| NEEDS_CONTEXT | 缺信息 | 提供信息，重新派发 |
| BLOCKED | 无法完成 | 评估阻塞原因，分类处理 |

**BLOCKED 处理：**
1. 缺上下文 → 提供更多，重新派发（相同模型）
2. 需要推理 → 重新派发（更强模型）
3. 任务太大 → 拆成小块
4. 计划错误 → 升级给人类

## 禁止事项

**永远不要：**
- ❌ 在主分支上直接开始实现（除非用户明确同意）
- ❌ 跳过 review（spec 合规或代码质量）
- ❌ 有未修复问题继续
- ❌ 并行派发多个实现子代理（会冲突）
- ❌ 让子代理自己读计划文件（直接提供完整文本）
- ❌ 跳过场景设置（子代理需要知道任务上下文）
- ❌ 忽略子代理的问题
- ❌ "差不多就行"（审查发现问题 = 未完成）
- ❌ 跳过 review 循环（发现问题 = 修复 = 重新审查）

## 子代理派发模板

### 实现代理
提供：
- 任务完整文本（从计划中提取）
- 上下文（任务在整体计划中的位置）
- 约束（技术限制、编码规范）
- 命令格式和提交流程

### Spec 审查代理
检查：
- 代码是否符合 spec 要求
- 没有多做（spec 没有的功能）
- 没有少做（spec 有的功能没实现）

### 代码质量审查代理
检查：
- 代码质量、测试覆盖
- 是否有明显 bug
- 是否遵循最佳实践

## 执行示例

```
你：我正在使用 subagent-driven-development 执行此计划。

[读取计划文件一次]
[提取所有任务完整文本和上下文]
[为所有任务创建 TodoWrite]

Task 1: Hook 安装脚本

[获取 Task 1 文本和上下文]
[派发实现子代理，提供完整信息]

实现者：在开始前 — hook 应该安装在用户级别还是系统级别？

你：用户级别（~/.config/openclaw/hooks/）

实现者：明白。正在实现...
[实现完成]
- 实现了 install-hook 命令
- 添加测试，5/5 通过
- 自审：发现缺少 --force 标志，添加了
- 提交

[派发 spec 合规审查]
审查者：✅ Spec 合规 - 所有要求满足，无额外功能

[派发代码质量审查]
审查者：优点：测试覆盖好，代码干净。问题：无。批准。

[标记 Task 1 完成]

Task 2: 恢复模式

...

[所有任务完成后]
[派发最终代码审查]
最终审查：所有要求满足，可以合并

完成！
```

## 优势

**vs 顺序执行：**
- 子代理自然遵循 TDD
- 新鲜上下文（无混淆）
- 并行安全（子代理不互相干扰）
- 子代理可以提问（过程中也可以）

**vs 单次执行：**
- 同一会话（无切换开销）
- 持续进展（无等待）
- 自动 review 检查点

**质量保证：**
- 自审在交接前捕获问题
- 两阶段 review：spec 合规 → 代码质量
- Review 循环确保修复有效
- Spec 合规防止多做/少做

## 整合

**必需的工作流 skills：**
- `brainstorming` — 创建 spec
- `writing-plans` — 创建此 skill 执行的计划
- `finishing-a-development-branch` — 所有任务后的收尾

**子代理应该使用：**
- `test-driven-development` — 每个任务遵循 TDD

**替代方案：**
- `executing-plans` — 用于并行会话而非同会话执行
