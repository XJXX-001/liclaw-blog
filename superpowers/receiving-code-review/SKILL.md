# Receiving Code Review

## 触发条件

收到代码审查反馈（来自子代理或人类）时使用。

## 接收反馈

### 记录反馈

收到审查后，先记录所有问题：

```markdown
## 审查反馈

### 问题列表
- [ ] 问题 1（[严重/中等/轻微]）
- [ ] 问题 2（[严重/中等/轻微]）
- [ ] 问题 3（[严重/中等/轻微]）

### 结论
[通过/修复后通过/需要大改]
```

## 问题分类

### 严重问题（必须修复）

- 安全漏洞
- 正确性 bug
- 数据丢失风险
- 重大性能问题

### 中等问题（应该修复）

- 代码不清晰
- 测试覆盖不足
- 规范违规
- 潜在问题

### 轻微问题（可以考虑修复）

- 风格偏好
- 命名优化
- 小优化
- 注释改进

## 处理流程

```
收到审查反馈
    ↓
分类所有问题
    ↓
修复严重问题
    ↓
修复中等问题
    ↓
评估轻微问题（修复或解释）
    ↓
提交修复
    ↓
回应审查
```

## 修复指南

### 每类问题的处理

| 类型 | 处理 |
|------|------|
| 严重 | 必须修复，修复后重新审查 |
| 中等 | 应该修复，修复后重新审查 |
| 轻微 | 修复或提供理由不修复 |

### 修复原则

- **一次只修一个问题**
- **修复后测试**
- **保持提交粒度小**

## 回应审查

### 当修复问题时

```markdown
## 审查回应

已修复以下问题：

### 问题 1（严重）
修复：...
验证：测试通过

### 问题 2（中等）
修复：...
验证：代码已改善

请重新审查。
```

### 当不采纳建议时

```markdown
## 审查回应

关于问题 X：

解释：为什么当前实现是合理的

[如果提供替代方案]
替代考虑：...

请评估。
```

## 常见情况处理

| 情况 | 处理 |
|------|------|
| 审查超时 | 提醒审查者，或升级给人类 |
| 审查者有分歧 | 升级给人类仲裁 |
| 审查过于严格 | 解释你的设计决策 |
| 审查建议不合理 | 提供数据和理由 |

## 结束标志

- 所有严重问题已修复 ✅
- 所有中等问题已修复或达成共识 ✅
- 轻微问题已评估 ✅
- 审查者确认通过 ✅
